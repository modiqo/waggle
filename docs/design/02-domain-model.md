# 02 — Domain Model

*The objects, their exact meanings, and the invariants that hold everywhere.
Types are shown in Rust but read this document as vocabulary, not API — the
API surface lives in 09.*

> **Revision 2 note.** The domain model survived the deep-research pass and
> the MCP-first pivot unchanged — it is transport-independent by construction.
> Two additions only: the primary consumption interface is now the MCP tool
> triplet (`mint` / `resolve` / `record` — 09 §3), whose arguments and results
> are exactly the objects on this page; and `Event` gains an optional
> `variant: Option<u8>` field recording which manifest variant served a
> resolution (ratified against I-1: it references the manifest's own content,
> never recipient data — settles 06's open question #2).

## 1. The two-object heart

Everything reduces to two objects with opposite lifecycles:

| | **CanonicalUrl** | **Token** |
|---|---|---|
| is | permanent identity of the target | one act of distribution |
| cardinality | one per artifact | many per artifact (one per share) |
| mutability | never mutated by sharing | manifest has versioned mutable sections |
| accrues | the aggregate story (all funnels) | one channel's funnel |
| dies | never (tombstones, not deletion) | revocable, expirable |

Keeping identity and distribution as *separate objects* is the design's first
decision and everything else follows from it.

## 2. Object inventory

### Token

A short, non-enumerable name. Base58 alphabet (Bitcoin variant — no `0OIl`
ambiguity), default 8 characters (58⁸ ≈ 1.28 × 10¹⁴), generated by rejection
sampling so there is no modulo bias. Fixed-size inline storage (03 §4) — a
`Token` is `Copy`, 24 bytes, never heap-allocates.

### AttributionManifest

The stored, retrievable record behind a token. Three zones with three
mutability rules:

```text
AttributionManifest
├── immutable core                     (fixed at mint, signable)
│     schema: u16            — envelope version, pinned at 1
│     token: Token
│     target: CanonicalUrl
│     sharer: Sharer         — who minted; independent of authorship
│     channel: Channel       — where this share lives
│     minted_at: Timestamp
│     meta: TargetMeta       — mint-time snapshot: title, description,
│                              image URL, author handle, labels
├── variants: Vec<Variant>             (fixed at mint in v1; see [open])
│     match_: MatchExpr      — which resolver contexts this serves
│     body: VariantBody      — the representation for that consumer
└── mutable sections                   (versioned; every change is an event)
      campaign: Option<CompactString>
      labels: SmallMap<Label, Value>
      expires_at: Option<Timestamp>
      revoked_at: Option<Timestamp>
      superseded_by: Option<Token>     — repaired/updated pointer
```

**Rule: `TargetMeta` is a snapshot supplied by the minter, never scraped.**
The system performs no requests against targets — unfurls are deterministic
and the minter controls their own representation.

**[open]** Whether variants may be *added* post-mint (as manifest-mutation
events) or are strictly mint-fixed. v1: mint-fixed; the event model (04)
already supports relaxing this without a schema break.

### MediaRef (rev 2.3 — images, voice, binary artifacts)

Bytes never ride the log (I-1 and the wire format both forbid it). A variant
body or target is either **inline** (small text/data, ≤ ~64 KB — the range
where SQLite reads beat the filesystem) or a **MediaRef**:

```text
MediaRef { uri: CanonicalUrl,        — where the bytes live (CAS, 07 §4)
           content_type: CompactString,
           size: u64,
           sha256: [u8; 32] }        — integrity: verify-what-you-fetched
```

Rules: the inline/ref threshold is automatic (the caller never chooses
wrong); total manifest size is capped (~256 KB); delivery is out-of-band
(resolve returns the URL + hash; bytes never stream through an MCP tool
response — the ≤4 KB budget survives); `mint --attach <file>` copies bytes
into the content-addressed store and mints the MediaRef. Multimodal
consumers are served by the existing variant machinery — `ModalitySet`
already carries vision/audio, so one token offers the image to the
vision-capable agent, the voice note to the audio agent, and an inline
transcript to the catch-all (worked example in 06 §2).

### Channel

An open slug taxonomy (`x-twitter`, `slack`, `linkedin`, `email`, `qr-event`,
`newsletter`, `subagent/<role>`, …). Well-known constants provided;
custom slugs validated (`[a-z0-9._/-]{1,64}`, lowercased). One token per
channel is enforced by API shape: `mint` takes exactly one channel;
`mint_batch` is the multi-channel convenience that returns one token each.

### Sharer

Who performed the distribution act. A validated slug, deliberately *not* an
authenticated identity in the core (hosts bind sharers to their own auth).
The sharer is first-class because distribution deserves credit independent of
authorship — the person who forwards a link into the right room did work.

### Stage & Event

```text
Stage      — interned identifier; well-known set pre-interned:
             impression · click · resolve · assess · consent ·
             install · signin · credential · run · repeat
             + open custom stages (validated slugs)

Event      — { token, stage, actor: ActorClass, at: Timestamp, seq: Seq,
               variant: Option<u8> }   — which manifest variant served a
             resolve; manifest-referencing, so I-1-compatible.
             THERE IS NO PAYLOAD FIELD. This is invariant I-1.

ActorClass — Bot | Human | Terminal | Agent { model_family, harness }
             — coarse classes only; model_family/harness are enumerated
             families (e.g. "claude", "gpt", "gemini", "other"), never
             versions, instance IDs, or fingerprints.
```

The stage vocabulary being *open* is what extends the funnel past the click:
the consuming product reports its own lifecycle stages against the token.
This is the capability no shortener has.

### ResolverContext

The consumer's declared context at resolution time. Sourced from an **A2A
Agent Card** (the adopted schema — see 06) via a pluggable extractor, or from
UA negotiation for non-agent consumers:

```text
ResolverContext
    kind: ConsumerKind         — Bot | Human | Terminal | Agent
    model_family: Option<Family>
    harness: Option<Harness>
    modalities: ModalitySet    — bitset: text · browser · shell · vision …
    posture: Posture           — Attended | Headless | Ci
```

### Resolution

What resolve returns: `{ disposition, projection, as_of, revalidate_after }`
where

```text
Disposition — Active | Expired | Revoked { at } | Superseded { by }
Projection  — the 301 target (human) | unfurl meta (bot) |
              text card (terminal) | selected Variant body (agent)
as_of       — the Timestamp this resolution reflects (point-in-time, always)
revalidate_after — advisory freshness window (variant-configurable; short
              for sensitive artifacts): "re-resolve before acting if older"
```

A resolution is knowledge, not a lease — waggle cannot revoke what a
consumer already holds. `as_of`/`revalidate_after` make that contract
explicit and give agents the re-check affordance (G-3, 15 §3 A6). Policy
(redirect anyway on Expired? tombstone page shape?) belongs to hosts; the
core returns facts.

Mutations carry concurrency semantics (G-4, 15 §3 B3): **lifecycle**
changes (`Revoked`, `Superseded`, `ExpirySet`) require `expected_version`
and fail with `Conflict` on mismatch (compare-and-set); **cosmetic** changes
(campaign, labels) are last-writer-wins by commit order. Minting carries a
client `mint_nonce` making it idempotent under retry (G-5): a retried mint
returns the original token, never a duplicate.

### Lineage

Tokens form a forest: a token minted *from* another token records
`parent: Option<Token>`. An orchestrator handing work to a subagent re-mints
a child token (`channel: subagent/<role>`); the resulting tree is
simultaneously the delegation trace, the cross-swarm attribution graph, and
the revocation cascade path (06 §4).

## 3. The invariants (numbered; tests cite these)

- **I-1 · No payload.** `Event` carries token, stage, actor class, time, seq —
  nothing else. Recipient data cannot enter analytics *by construction*.
- **I-2 · Deterministic projection.** `select_variant(manifest, ctx)` is a
  pure total function: same manifest version + same context → same variant,
  every time, on every platform. Adaptivity must be auditable.
- **I-3 · Snapshot metadata.** Unfurl content derives only from mint-time
  `TargetMeta`. The system never fetches targets.
- **I-4 · Resolution never blocks on attribution.** The redirect/projection
  is computed from the manifest alone; event recording is a separate,
  asynchronous act. (API shape enforces: `resolve` doesn't take a store
  writer.)
- **I-5 · The log is the truth.** Every counter, funnel, and report is a fold
  over the event log; manifest mutations are themselves events; therefore any
  historical state or statistic is reconstructable (04).
- **I-6 · Identity outlives distribution.** Canonical URLs are never rewritten;
  revoked tokens tombstone (never 404-recycle); token names are never reused.
- **I-7 · Coarse actors only.** `ActorClass` is the *maximum* granularity the
  event log can hold. Attributed/signed resolution (v2) is opt-in and lives
  beside, not inside, the event log.

## 4. Vocabulary discipline

The words are load-bearing and used consistently across all docs and code:
**mint** (create token+manifest), **resolve** (consumer → disposition +
projection), **record** (append an event), **unfurl** (bot-facing projection),
**variant** (agent-facing representation), **funnel** (stage-count fold),
**reconstruct** (replay events to rebuild any state or statistic),
**lineage** (parent/child token forest), **tombstone** (revoked-but-present).
